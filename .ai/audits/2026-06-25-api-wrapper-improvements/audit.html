AniList Wrapper Improvement Audit

TL;DR

The wrapper is already in a good shape for a small generated GraphQL package: it has a clear public entrypoint, service classes over generated operations, docs, examples, build scripts, and a raw GraphQL escape hatch. The next improvement should not be a rewrite. The best first task is to add deterministic, non-network contract tests around the client and service variable mapping, then tighten public API naming and option consistency behind backwards-compatible aliases.

After the contract layer is protected, add a small shared pagination/filter input pattern and expand convenience methods only where current docs/examples force users back to raw GraphQL for common AniList workflows.

1. Goal

Improve the API wrapper's reliability, maintainability, and developer ergonomics while preserving the existing public API and generated AniList response shapes.

2. Codebase Understanding

  • Stack: TypeScript ESM package, Bun scripts/tests, tsdown build, Biome checks, GraphQL Code Generator, and @api-wrappers/api-core transport.
  • Package surface: package.json:43 exports ESM/CJS/types from dist; package.json:60 defines build, codegen, test, typecheck, check, and verify scripts.
  • Main API: src/index.ts:16 exposes an Anilist class that creates one generated SDK client and service instances for anime, manga, characters, staff, users, media, media lists, and raw GraphQL.
  • Transport: src/client/index.ts:7 builds a GraphQL client with JSON headers, optional bearer token, retry for HTTP 429, and a local fragment deduplication pass at src/client/index.ts:46.
  • Services: service methods mostly forward typed variables into generated SDK operations, for example src/services/animeService.ts:32, src/services/mangaService.ts:35, src/services/userService.ts:42, and src/services/mediaListService.ts:78.
  • Docs: README already positions the package well, with raw GraphQL escape hatch guidance at README.md:44 and generated type maintenance notes at README.md:221.
  • Existing roadmap: ROADMAP.md:26 already calls out media-list docs, examples, naming consistency, request-construction tests, and codegen notes. This audit agrees with that direction but prioritizes the test/contract work first.
  • Tests: current tests are mostly live AniList integration tests with one-second delays and rate-limit skips, visible in test/anime.test.ts:11 and test/utils.ts:32. They verify broad runtime behavior but do not reliably protect service variable mapping or transport options.

3. Recommendation

Do this now, in two execution tasks.

Task one should add deterministic contract tests and tiny test-only helpers for the client/service seams. This gives future changes a safety net without changing behavior. Task two should use that safety net to normalize public API ergonomics with backwards-compatible aliases and shared option types. Do not replace the generated SDK layer, do not wrap AniList responses into custom model classes, and do not rename existing methods without aliases.

4. Scope

In Scope

  • Add deterministic tests for service method variable mapping.
  • Add deterministic tests for raw GraphQL request forwarding and transport behavior.
  • Add backwards-compatible aliases for inconsistent public method names.
  • Add small shared input/helper types for pagination, media type normalization, and common browse filters where they reduce duplication.
  • Update docs and examples to show the preferred method names and option patterns.

Out of Scope

  • No rewrite of the generated SDK or GraphQL Code Generator setup.
  • No custom domain model layer over AniList response shapes.
  • No breaking method renames or response-shape changes.
  • No dependency additions unless a test seam cannot be covered with Bun's existing test tools.
  • No schema migration or persistence work; this package does not own data storage.

5. Files and Areas to Inspect

  • src/client/index.ts for transport, auth, retry, signal, request headers, and fragment deduplication.
  • src/index.ts for public exports and class construction.
  • src/services/*.ts for method naming, input types, and SDK variable mapping.

  • codegen.yml and scripts/patch-codegen.ts only if GraphQL operations or generated SDK behavior changes.

6. Implementation Plan

  1. Create a non-network service contract test harness. Build a typed fake SDK object that records calls and returns minimal resolved values. Use it to instantiate each service class directly and assert exact operation names and variables for representative methods.
  2. Cover the highest-risk service mappings first. Test AnimeService, MangaService, UserService, MediaService, and MediaListService. Include default page/perPage/status behavior and media type conversion.
  3. Add raw GraphQL service tests. Instantiate GraphQLService with a fake GraphQLClient and assert document, variables, request headers, and signal forwarding.
  4. Add transport/deduplication coverage. Either export a narrowly named internal helper from src/client/index.ts only if acceptable for tests, or test through createGraphQLClient by mocking @api-wrappers/api-core. Cover auth token header, custom request headers, retry config, duplicate fragment removal, nested braces, and malformed fragment definitions.
  5. Separate live smoke tests from deterministic tests. Keep existing AniList live tests, but make it clear they are integration smoke coverage. If the project wants CI determinism, introduce a script such as test:unit for non-network tests and make verify depend on that first.
  6. Add backwards-compatible naming aliases. Add preferred aliases for favorite mutations while preserving toggleFavourite. The preferred API should be documented, but existing callers must continue to work.
  7. Add small shared input types for future ergonomics. Define exported types such as PaginationOptions and service-specific filter objects where the wrapper already accepts filters. Do not convert every existing positional method at once unless overloads preserve compatibility.
  8. Normalize media type conversion internally. Replace repeated string-to-enum conversions with a tiny helper. Keep accepting the existing MediaTypeNonEnum input unless a generated enum overload can be added without breaking callers.
  9. Update docs and examples after code changes. Update README, API docs, and examples to show preferred aliases, option patterns, auth requirements, and deterministic testing guidance for contributors.
  10. Run package validation. Run check, typecheck, unit tests, build, and package dry run. Run live AniList smoke tests separately when network and rate limits permit.

7. Code Quality Rules

  • Follow existing TypeScript, Bun, Biome, and tsdown patterns.
  • Keep all changes backwards compatible unless a major-version plan is explicitly created.
  • Do not use any. If generated code contains any, do not spread it into handwritten code.
  • Do not introduce a custom object model over AniList GraphQL responses.
  • Reuse generated operation types and schema enums.
  • Add small helpers only where duplication is already repeated across services.
  • Keep test fakes local and typed; avoid weakening the service constructor types with broad casts.

8. Testing and Validation

  • bun run check
  • bun run typecheck
  • bun test --timeout 10000
  • bun run build
  • bun pm pack --dry-run
  • bun run verify after the focused checks pass.
  • bun run codegen only when GraphQL operation documents, fragments, or generated schema/type behavior changes.

9. Acceptance Criteria

  • Deterministic non-network tests cover representative service methods and exact SDK variable mapping.
  • Transport tests cover auth header setup, retry config, request header forwarding, abort signal forwarding, and fragment deduplication behavior.
  • Existing public method names still work.
  • Preferred aliases and option types are documented and exported where useful.
  • No generated files are edited by hand.
  • Package validation commands pass, or any live-network failures are clearly separated from deterministic unit failures.

10. Risks and Notes

  • Compatibility risk: method renames can break consumers. Use aliases first, document preferred names, and defer removals to a major-version plan.
  • Test design risk: over-mocking can make tests brittle. Assert stable wrapper contracts: operation names, variables, request options, and public return passthrough.
  • Generated-code risk: codegen output can contain broad generated types. Keep handwritten wrapper code strict and do not patch generated files manually outside the existing script.
  • Network risk: live AniList tests can fail due to rate limits or schema/data changes. Treat them as smoke tests, not the only correctness gate.
  • Scope risk: adding many convenience methods at once will create a larger support surface. Add methods only when docs/examples demonstrate common workflows.

Recommended Sequencing

  1. First task: deterministic contract tests for services and client behavior.
  2. Second task: backwards-compatible API ergonomics cleanup and docs updates.
  3. Third task: new convenience methods driven by examples and issue requests.

Do not start with a broad abstraction or rewrite. The current architecture is good enough; it needs guardrails and polish.
